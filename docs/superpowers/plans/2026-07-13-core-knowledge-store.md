# Core Knowledge Store Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build delivery slice 1: a transactional SQLite knowledge store with project registration, deterministic alias resolution, append-only events, and validated acyclic Case graphs.

**Architecture:** A small TypeScript library owns the SQLite connection and exposes repositories rather than leaking SQL to future MCP and CLI adapters. Every mutation runs through one transaction that updates materialized tables and appends an immutable event. Domain validation is kept independent from persistence so graph rules can be tested without a database.

**Tech Stack:** Node.js 22, TypeScript 5, better-sqlite3, Vitest, npm

## Global Constraints

- Store runtime data only under `data/`; Git must ignore it.
- Enable SQLite WAL mode and foreign keys on every writable database connection.
- Every mutation must append an event and update materialized state in one transaction.
- Project-scoped operations must resolve `project_id` or `project_root` deterministically and reject ambiguity.
- Historical events are append-only; graph history must not be rewritten.
- Relations inside a Case must remain acyclic.
